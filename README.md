# Мини-облачный провайдер на FastAPI

REST API для управления виртуальными дисками с поддержкой файловых систем ext4, xfs и btrfs.

---

## О проекте

Проект вырос из лабораторной работы по операционным системам.  
В процессе изучения файловых систем в Docker я заинтересовалась тем, как устроены облачные хранилища, и решила написать свой мини-провайдер.

Идея простая: автоматизировать создание, форматирование и монтирование дисков через HTTP-запросы.  
В результате получился сервис, который умеет всё то же самое, что и настоящие облачные провайдеры, только в миниатюре.

---

## Технологии

- **Python** + **FastAPI** — основа бэкенда
- **Pydantic** — валидация входящих данных
- **Docker** + **Docker Compose** — контейнеризация
- **Linux-утилиты**: `dd`, `mkfs`, `mount`, `umount`
- **subprocess** — запуск системных команд из Python

## Что умеет API

| Метод | Эндпоинт | Что делает |
|-------|----------|------------|
| GET | `/` | Проверка, что сервер жив |
| GET | `/disks` | Список всех дисков |
| GET | `/disks/{name}` | Данные конкретного диска |
| POST | `/disks/{name}` | Создать новый диск |
| POST | `/disks/{name}/mount` | Примонтировать диск |
| POST | `/disks/{name}/umount` | Размонтировать |
| DELETE | `/disks/{name}` | Удалить диск |
| GET | `/metrics` | Статистика использования |

## Как работают диски

Каждый виртуальный диск — это обычный файл с расширением `.img`, который лежит в папке `disks_images/`.

При создании диска происходит два шага:
1. **`dd`** — создаётся пустой файл нужного размера, заполненный нулями
2. **`mkfs`** — файл форматируется в выбранную файловую систему

Пользователь получает диск, который можно монтировать как настоящий.

---

## Поддерживаемые файловые системы

| ФС | Особенности |
|----|-------------|
| **ext4** | Универсальная, работает с любым размером |
| **xfs** | Для больших нагрузок, **минимальный размер — 300 МБ** |
| **btrfs** | Современная, с продвинутыми функциями |

## Идемпотентность

Одно из важных требований к хорошему API — возможность повторять запросы без вреда.

В проекте это реализовано так:

- **Создание диска** — если диск с таким именем уже есть, API не выдаёт ошибку, а просто возвращает существующий диск.  
  Это полезно, если клиент случайно отправил запрос дважды.

- **Удаление диска** — если диск уже удалён, при повторном запросе API всё равно отвечает успехом (с сообщением, что диск уже удален).  
  Клиенту не нужно обрабатывать ошибку.

## Детали конфигурации

### `docker-compose.yml`

Файл `docker-compose.yml` описывает, как должен запускаться проект:

— образ собирается из текущей папки (`build: .`);
— контейнеру выданы права на монтирование дисков (`privileged: true`);
— порт `8000` открыт для доступа к API;
— папка `./disks_images` на компьютере подключается внутрь контейнера, чтобы диски не пропадали после остановки.

Благодаря этому всё необходимое для работы уже описано в файле — запуск сводится к одной команде `docker-compose up`.

## Запуск в Docker

Проект использует Docker не просто так. Именно благодаря контейнеру мы можем:

- **Работать с разными файловыми системами** — внутри контейнера Linux, где есть все нужные утилиты (xfsprogs, btrfs-progs)
- **Запускать проект где угодно** — на Mac, Windows или Linux, независимо от того, что установлено в системе
- **Не захламлять компьютер** — всё живёт внутри контейнера
- **Быстро разворачивать** — образ лёгкий, запуск занимает секунды

Для запуска в терминале выполнить:
```bash
docker-compose up
```
После запуска сервис будет доступен по адресу http://localhost:8000/docs — там находится Swagger UI для тестирования API.
